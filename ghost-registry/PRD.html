Ghost Registry (Ghosty)

A decentralized, unstoppable software registry foundation.

1. Vision & Mission

Ghost Registry is designed to eliminate platform risk in the global software supply chain. It leverages the Freenet (Locutus) protocol to ensure that once a piece of code is published, it remains available and verifiable forever.

2. Problem Statement

  • Platform Centralization: Registries like NPM and PyPI are controlled by single entities with the power to delete or block content.
  • Supply Chain Fragility: The "left-pad" incident showed how one missing dependency can break millions of systems.
  • Planned Obsolescence: Repositories often disappear when companies shut down, leaving old projects unbuildable.

3. Core Features (Phase 1)

3.1. Immutable Hosting

Packages are stored on the Freenet P2P network, addressable by cryptographic hash (CID).

3.2. Cryptographic Authorship

Developers sign updates with private keys; the network verifies signatures before updating version pointers.

3.3. Decentralized Versioning

Uses Freenet state management to track SemVer versions globally without a central coordinator.

3.4. Proxy Compatibility Layer

A local CLI daemon acts as a proxy for existing tools (e.g., npm config set registry http://localhost:8080).

4. Technical Architecture

  • Protocol: Freenet (Locutus).
  • Logic: WASM Contracts (running the registry rules).
  • Storage: Distributed Hash Table (DHT).

5. Open Questions

How do we handle name squatting without a central moderator? How do we incentivize nodes to store large binaries? These are the puzzles we'll solve next.
